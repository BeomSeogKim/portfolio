# eBay 글로벌 상품 연동 시스템 구축 및 아키텍처 고도화

> 2025.01 ~ 2025.09 · SFTP 연동 자체 구축, API 연동 핵심 로직 개발, TypeScript 이식 및 아키텍처 고도화

`Java` `Kotlin` `TypeScript` `Node.js` `NestJS` `Redis` `Kafka`

**약 40만 개 상품 기준 동기화 시간 24시간 → 3시간 (8배 단축), 글로벌 채널 25년도 흑자 전환의 핵심 기반**

<br/>

## Background

- 기존에 eBay 글로벌 마켓플레이스에 사내 중고 상품 및 일부 파트너사 새상품을 수기로 등록하는 방식으로 운영
- 운영 확장성 한계로 자동화 필요, 사내 eBay 도메인 지식 부재 상태에서 프로젝트에 자원

<br/>

## Phase 1 — SFTP 연동 자동화 플로우 자체 구축

- 리세일팀과 협력하여 eBay SFTP 연동 자동화 시스템을 Zero-to-One으로 설계 및 개발
- eBay Developer 문서 학습 및 eBay 기술지원팀과 직접 커뮤니케이션하며 도메인 지식 습득
- 예상 이상의 해외 판매 성과로 API 연동 전환 결정

## Phase 2 — API 연동 전환 및 핵심 변환 로직 개발

- 빠른 적용을 위해 상품팀과 공동 작업: 상품팀은 NestJS 인프라 구축, Java로 핵심 로직 개발 담당
- Trenbe 상품 → eBay 규격 변환, 6개국 카테고리 매핑, 가격/재고 동기화 등 비즈니스 규칙 전체 구현

## Phase 3 — 기술 스택 통합을 위한 TypeScript 이식

- 핵심 로직이 Java로 개발되어 상품팀(TypeScript)과 기술 스택 불일치 및 유지보수 어려움 발생
- 2~3주 TypeScript 학습 후 Node.js 환경으로 직접 이식, 운영 포인트 단일화

<br/>

## Architecture

### 국가 확장성을 고려한 헥사고날 아키텍처 도입

- 6개국 연동 중 국가별로 상이한 변환 로직과 공통 인프라가 강하게 결합된 구조
- 외부 의존성을 어댑터로 분리, 핵심 로직을 도메인 영역으로 격리하여 단위 테스트 커버리지 100%
- 국가 추가 시 Router 큐 설정 + Consumer 클래스 1개만 추가하면 되는 확장 구조

### Router 패턴 기반 멀티 채널 분산 처리

- 단일 큐에서 6개국 요청을 순차 처리 → 국가별 전용 큐로 분산
- Promise 기반 비동기 병렬 처리로 국가 수 증가에도 선형 성능 저하 없이 독립 처리

### 토큰 발급 동시성 처리 개선

- 다수 상품 동시 연동 시 Access Token 만료되면 모든 요청이 순차 대기하는 병목 발생
- 진행 중인 토큰 발급 요청의 Promise 객체를 공유하여 중복 호출 제거
- Exponential Backoff 및 Cooldown 전략으로 API Rate Limit 대응

<br/>

## 연동 로직 고도화

- 입점 파트너사의 부정확한 카테고리 등록으로 eBay Title 기반 검증에서 대량 연동 에러 발생
- 1단계: 강제 매핑 → eBay Decision API 연동으로 전환, 대량 처리 시 Rate Limit 병목에 직면
- 2단계: Redis 기반 다계층 캐싱 시스템 설계 및 도입
  - 상품명에서 핵심 키워드를 추출·정규화·정렬하여 캐시 히트율 극대화
  - 데이터 변경 빈도에 따라 가변 TTL 적용
- **상품 연동 일일 실패율 99.5% 감소** (평균 2,000건 → 10건 미만)
- **eBay 상품 연동 성공률 약 1.7배 증가** (일 평균 6,000건 → 10,000건)

<br/>

## 글로벌 연동 프로세스 분리

- Shopify 연동 서비스가 번역, 이미지 정제, 상품 변경 이벤트 발행 등 과도한 역할을 담당
- 글로벌 연동 서비스가 Shopify 연동 조건에 의존하여 신규 채널 추가 시 유연성 저하
- 공통 번역 서비스를 상위로 분리, 번역 완료 후 각 글로벌 연동 서비스에 이벤트 발행하는 구조로 변경
- 신규 글로벌 채널 추가 시 Shopify 의존 없이 독립 연동 가능한 구조로 전환

## 통합 데이터베이스 스키마 설계

- 글로벌 채널 연동에 필요한 데이터가 채널별로 분산 관리되어 운영 복잡도 증가
- 공통 데이터 중앙화 + 채널별 메타데이터 분리 + 이력 관리 통합 구조로 설계
- 하나의 테이블에서 모든 글로벌 채널별 연동 성공/실패 여부, 에러 내역을 추적 가능
